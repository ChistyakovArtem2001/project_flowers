# Детекция декоративных цветов и растений на фасадах

Проект по дисциплине **«Сбор, генерация и разметка данных»**: детекция декоративных цветов и растений на фотографиях фасадов заведений (Москва). Разметка в формате **YOLO** (один класс `flower`), изображения - дневные/вечерние/ночные снимки.

## Что лежит в этом репозитории

Здесь выложена **учебная подвыборка** (мини-датасет), на которой обучалась модель для **полуавтоматической** разметки остальных кадров. Полный продовый датасет (≈4,5k фото, 20k+ объектов) в репозиторий не входит из соображений объема.

## Формат разметки

Каждая строка в `*.txt`:

```text
<class_id> <x_center> <y_center> <width> <height>
```

Все координаты **нормированы** на ширину и высоту изображения (0…1). Для этого датасета `class_id` всегда `0` (класс `flower`).

## ТЗ для разметки и разметчиков

Ниже правила для **bbox** прямоугольников (YOLO): модель видит только прямоугольники, поэтому важно, как именно они «ложатся» на невыпуклые и разветвленные композиции.

### Что размечать

- **Декоративные живые растения и цветы** на фасаде заведения: ящики и клумбы у входа, вертикальное озеленение, арки и порталы из зелени/цветов, венки и гирлянды **из реальной зелени**, стелющиеся или каскадные посадки на вывеске/витрине, если это именно **ботанический декор**, а не только картинка.
- Размечается **видимая** часть: если растение обрезано кадром или перекрыто, bbox охватывает только то, что реально видно (без «догадки» о продолжении за краем кадра).

### Что не размечать

- Одна только **печатная/нарисованная** зелень на баннере, наклейке или экране без живых растений рядом.
- Горшки/ящики **без** видимой зелени (пустые контейнеры).
- Люди, транспорт, мусор, не относящиеся к декору; сильно размытые пятна, в которых нельзя уверенно сказать «это растения».

При сомнении «растение или декор» - **не включать** или согласовать.

### Как ставить прямоугольник

- **Плотный bbox**: стороны прямоугольника касаются краев видимой кроны/массы растений, без больших полей «воздуха» внутри рамки и без обрезки заметной части зелени.
- Один физический куст/ящик, который визуально **цельный** и компактный, — **один** bbox.
- Два **отдельных** декоративных блока (две клумбы в разных углах, два ящика с разрывом) — **два** bbox, даже если класс один.

### Сложная форма: когда бить на несколько объектов

Детектор использует **прямоугольники**. Если одна зеленая конструкция **сильно не выпуклая** или **разветвленная**, один большой прямоугольник дает много пустой площади внутри и путает обучение (модель учится на «фон внутри рамки»).

**Нужно разбивать на несколько bbox**, если форма похожа на:

- **Букву П, Г, L, подкову, арку с опорами**: отдельно вертикальные/наклонные **стойки** и отдельно **верхняя часть** (дуга, перекладина, «перемычка»), если между ними заметный **разрыв по фону** или изгиб такой, что один прямоугольник покрывает слишком много стены без растений.
- **Кольцо или овал** из зелени: разумно **2–4 перекрывающихся или стыкующихся** прямоугольника по дуге/четвертям, чтобы суммарно покрыть зелень без огромной «дыры» в центре одного bbox.
- **Длинная вертикальная стена** из растений с явными **провалами** (окно, вывеска, пустой участок стены) — **несколько bbox по вертикальным сегментам**, разделенные там, где зелень реально прерывается.

**Не дробить без нужды:** если композиция **плотная «подушка»** или один ящик без сильных выступов - остается **один** bbox.

Практическое правило: если при мысленной подстановке **одного** минимального прямоугольника **больше ~25–35%** площади рамки занимает **не растения** (стена, стекло, пустота), имеет смысл **разделить** на два или больше прямоугольников по логичным частям конструкции.

### Перекрытия и границы

- Рамки **могут пересекаться** на стыке частей одной композиции — это нормально, если так лучше закрывается зелень без лишнего фона.
- Растение **перед** витриной: bbox по контуру зелени; не тянуть рамку на все стекло, если зелень занимает только часть.

### Единообразие

- Одинаковые ситуации на разных фото размечать **одинаково** (тот же принцип дробления арок и буквообразных форм).
- Спорные случаи фиксировать.

## Авторазметка полного датасета

После того как на мини-выборке (этот репозиторий) была обучена первая версия детектора, оставшиеся ~3k фото размечались **полуавтоматически**: модель прогонялась по сырым кадрам, ее предсказания загружались в инструмент разметки как стартовые bbox, и разметчик уже **только правил** — удалял ложные срабатывания, дорисовывал пропуски и разбивал слипшиеся рамки по правилам из раздела «ТЗ для разметки». Так удалось не размечать ~3k фото с нуля.

### Чем делалось

Для прогона модели и подготовки готового архива для импорта в CVAT использовался [BelickNicko/AutoCVAT](https://github.com/BelickNicko/AutoCVAT) — небольшой CLI-обвес над Ultralytics, который:

- запускает инференс кастомных весов YOLO по папке с изображениями;
- собирает результат сразу в формате **COCO 1.0**, который CVAT принимает «из коробки» через `Actions → Upload annotations`;
- позволяет задать **per-class порог уверенности** и IoU для NMS через `config.yaml` — это удобно, потому что для класса `flower` приходилось загрублять порог, чтобы не плодить мусор на ночных кадрах.

Команда запуска выглядела примерно так:

```bash
python AutoCvat.py \
  --img_folder=raw_facades \
  --weights=runs/train/flowers_v1/weights/best.pt \
  --yaml_pth=config.yaml \
  --annotations_zip=flowers_pre
```

Дальше готовый `flowers_pre.zip` заливался в задачу в CVAT, и начиналась ручная доводка.

### Почему CVAT, а не Label Studio

Оба инструмента бесплатные и self-hosted, но под эту задачу CVAT подошел заметно лучше:

- **AutoCVAT отдает результат сразу в формате CVAT (COCO 1.0).** Чтобы пристроить тот же зип в Label Studio, пришлось бы писать конвертер в его внутренний JSON — лишнее звено в пайплайне, в котором и так уже хватает мест, где можно что-то перепутать.
- **CVAT изначально заточен под CV-разметку**, Label Studio — универсальный инструмент (текст, аудио, изображения, ранжирование). На плотных сценах с десятком bbox на кадр (а у нас именно такие — арки, П-формы, ряды ящиков) горячие клавиши и редактор рамок в CVAT работают ощутимо быстрее: дублирование, точная подгонка границ, скрытие/показ классов, навигация между объектами.
- **Импорт-экспорт YOLO нативный.** Финальный датасет нужен в YOLO-формате (`<class> <xc> <yc> <w> <h>`), и CVAT отдает его одной кнопкой. В Label Studio для YOLO нужно либо ставить конвертер, либо экспортировать в JSON и парсить отдельно.
- **ML-ассист без отдельного бэкенда.** В Label Studio для авторазметки прямо в интерфейсе нужно поднимать ML-backend (отдельный сервис, обертка модели в SDK Label Studio). AutoCVAT же работает «сбоку»: один CLI-вызов на машине с GPU, на выходе архив — никакой инфраструктуры рядом с инструментом разметки держать не надо.
- **Привычка команды.** В рабочем потоке CVAT уже использовался под другие CV-задачи, держать два разных инструмента под почти одинаковую работу смысла не было.

Label Studio имеет смысл, если задачи разметки разнородные (текст + изображения + аудио в одном проекте) или если нужны кастомные UI-сценарии. Для одной задачи детекции с одним классом эти преимущества не отыгрываются, а минусы (формат, скорость работы с bbox, отдельный ML-backend) бьют по делу каждый день.
